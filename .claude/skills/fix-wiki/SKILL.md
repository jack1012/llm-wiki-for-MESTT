# fix-wiki 技能 — 知識庫自動修復

根據 lint 巡檢結果，自動執行修復工作流，清理知識庫中的結構性問題（死連結、孤兒頁面、索引不同步）並更新全域登錄檔。

## 核心工作流：修復層 (Repair Layer)

fix-wiki 技能是 lint 的配套工具，完成以下修復流程：

1. **讀取 lint 報告** — 掃描知識庫找出問題
2. **自動修復**（可選）— 死連結移除、索引同步、孤兒頁面重連
3. **手動確認區**（可選）— 涉及內容刪除的決策留給使用者
4. **更新登錄檔** — 修改 index.md 和 log.md

## 觸發邏輯

1. **使用者執行 `/fix-wiki`**：執行所有建議修復（需逐項確認）
2. **使用者執行 `/fix-wiki <type>`**：執行特定類型的修復
   - `/fix-wiki orphan` — 修復孤兒頁面
   - `/fix-wiki deadlink` — 修復死連結
   - `/fix-wiki unsync` — 同步索引
   - `/fix-wiki conflict` — 處理知識衝突
3. **隱式觸發**：使用者說"修復知識庫"、"清理死連結"、"更新索引"時，自動執行 fix

## 修復流水線

### 步驟 1：診斷問題

執行內部 lint 邏輯或讀取最近的 lint 報告，識別待修復問題：

```
問題類型:
├── deadlink — 頁面中存在指向不存在檔案的 [[wikilink]]
├── orphan — 從未被任何其他頁面引用的孤兒頁面
├── unsync — 頁面存在但未在 index.md 中註冊（或反之）
└── conflict — 頁面包含 `## 知識衝突` 區塊，等待決策
```

### 步驟 2：分類修復策略

**A. 死連結 (deadlink) — 自動可修復**

**問題**：某頁面中有 `[[不存在的頁面]]`

**修復方式**：
1. 在頁面中搜尋死連結
2. 嘗試模糊匹配（如連結到 `[[RAG]]` 但實際頁面名為 `[[RAG架構]]`）
3. 如果無法模糊匹配 → 詢問使用者是否刪除該連結，或是否需要建立新頁面
4. 完成後更新頁面

**例子**：
```
❌ 存在死連結：[[非存在的概念]] 
✓ 修復方案A：刪除連結
✓ 修復方案B：建議建立新頁面 [[非存在的概念]]
```

**B. 孤兒頁面 (orphan) — 需要重連**

**問題**：某頁面從未被任何其他頁面引用

**修復策略**：
1. 檢查頁面標題和內容，識別其主題
2. 在其他相關頁面的 `## 關聯連結` 區域新增反向連結
3. 或建議在 index.md 中更新分類
4. 如果確實是過時/無用內容 → 詢問使用者是否刪除或歸檔

**例子**：
```
❌ 孤兒頁面：[[過時的概念]] 
   (從未被引用)

✓ 修復方案A：在 [[相關概念]] 頁面新增：
  ## 關聯連結
  - [[過時的概念]] — 歷史背景

✓ 修復方案B：確認過時，刪除此頁面
```

**C. 索引不同步 (unsync) — 自動同步**

**問題1**：檔案存在但未在 `wiki/index.md` 中註冊

**修復方式**：
1. 讀取頁面的 frontmatter，判定 `type`
2. 從頁面標題或 `title` 字段提取名稱
3. 在 index.md 相應分類下新增條目：
   ```markdown
   - [[PageName]] — 一句話描述（從頁面的第一句或摘要提取）
   ```

**問題2**：index.md 中註冊但檔案不存在

**修復方式**：
1. 標記為無效註冊
2. 詢問使用者：
   - A) 建立缺失的頁面？
   - B) 從 index.md 中移除此條目？

**D. 知識衝突 (conflict) — 需要人工決策**

**問題**：頁面包含 `## 知識衝突` 區塊，表示新舊知識有歧異

**修復方式**：
1. 讀取衝突記載
2. 展示兩種說法
3. 詢問使用者決策：
   - A) 保留兩者，標註為"待驗證"
   - B) 選擇其中一種，刪除另一種
   - C) 合併成新的綜合理解
4. 刪除 `## 知識衝突` 區塊，替換為統一觀點

### 步驟 3：執行修復與確認

**自動修復**（無需確認）：
- ✅ 索引同步：新增缺失的 index.md 條目
- ✅ 死連結：刪除或標記為"待驗證連結"

**需要人工確認**：
- ⚠️ 孤兒頁面：保留或刪除？
- ⚠️ 刪除頁面：確認前詢問
- ⚠️ 知識衝突：選擇最終立場

**確認流程**：
```
fix-wiki 技能向使用者展示：
1. 待修復項目清單
2. 對於需要確認的項，提供選項
3. 使用者確認後執行
4. 完成後生成修復報告
```

### 步驟 4：更新全域登錄檔

完成修復後，必須更新：

**更新 `wiki/index.md`：**
- 新增缺失的註冊條目
- 刪除不存在檔案的條目（已確認）
- 重新整理分類

**更新 `wiki/log.md`：**
追加操作日誌（Append-only）：
```markdown
## [YYYY-MM-DD] fix | 修復了 N 個問題
- **修復類型**: 
  - 死連結：M 個（已刪除或修復）
  - 孤兒頁面：K 個（已重連或保留）
  - 索引不同步：J 個（已同步）
  - 知識衝突：P 個（已決策）
- **變更**: 更新 [[index.md]], 修改 [[PageX]], [[PageY]], ...
- **備註**: [任何額外信息]
```

## 修復類型詳解

### deadlink 修復示例

**原始狀態**：
```markdown
---
title: "機器學習概論"
---

## 定義
基於 [[神經網路]] 和 [[深度學習框架]] 的技術...
但 [[深度學習框架]] 頁面不存在！
```

**修復過程**：
1. 發現死連結：`[[深度學習框架]]`
2. 搜尋類似頁面：找到 `[[深度學習]]`
3. 提議修復：
   ```
   ❓ 建議：將 [[深度學習框架]] 改為 [[深度學習]] 嗎？
   ```
4. 若確認 → 更新頁面
5. 若否認 → 詢問是否刪除連結或建立新頁面

### orphan 修復示例

**原始狀態**：
```
孤兒頁面：[[古老的計算機視覺方法]]
(從未被任何頁面引用)
```

**修復過程**：
1. 讀取頁面內容 → 主題是"傳統 CV 演算法"
2. 找到相關頁面：`[[計算機視覺]]`
3. 在 `[[計算機視覺]]` 頁面新增：
   ```markdown
   ## 關聯連結
   - [[古老的計算機視覺方法]] — 傳統方法，已逐漸被深度學習取代
   ```
4. 完成重連

### unsync 修復示例

**原始狀態**：
```
wiki/concepts/新概念.md 存在，但 index.md 未註冊

index.md:
## Concepts
- [[舊概念]] — 說明
(缺少 [[新概念]])
```

**修復過程**：
1. 掃描所有頁面 vs index.md
2. 發現未同步條目
3. 自動新增：
   ```markdown
   ## Concepts
   - [[舊概念]] — 說明
   - [[新概念]] — [自動提取的描述]  ← 新增
   ```

## 輸出與報告

完成修復後，生成結構化報告：

```markdown
## 🔧 fix 修復報告 — YYYY-MM-DD

### ✅ 已修復 (N 個)
- **死連結**: 3 個已刪除或修正
  - [[PageX]] 中的 [[不存在頁面]] → 刪除
  - [[PageY]] 中的 [[舊名稱]] → 改為 [[新名稱]]
- **孤兒頁面**: 2 個已重連
  - [[孤兒1]] ← 已在 [[父頁面]] 新增反向連結
  - [[孤兒2]] ← 已在 index.md 新增分類
- **索引不同步**: 4 個已同步
  - [[新頁面]] — 已在 index.md 註冊

### ⚠️ 待決策 (N 個)
- **知識衝突**: [[ConflictPage]]
  - 衝突雙方：說法A vs 說法B
  - 建議：[修復方案]

### 📊 修復統計
- 總計修復：10 個問題
- 自動修復率：70%
- 需要人工介入：30%
```

## 參數配置

| 參數 | 類型 | 說明 | 預設值 |
|------|------|------|--------|
| `type` | string | 修復類型：`all` \| `deadlink` \| `orphan` \| `unsync` \| `conflict` | `all` |
| `auto_confirm` | boolean | 是否自動執行所有可自動修復的項目 | `false` |
| `preserve_orphan` | boolean | 修復孤兒頁面時是否保留（而非刪除） | `true` |
| `report_format` | string | 報告格式：`brief` \| `detailed` | `detailed` |

### 使用範例

```
# 執行全部修復（需逐項確認）
/fix-wiki

# 僅修復死連結
/fix-wiki deadlink

# 自動執行所有可自動修復的項，跳過需要決策的項
/fix-wiki --auto_confirm

# 修復孤兒頁面，並保留（不刪除）
/fix-wiki orphan --preserve_orphan
```

## 與其他技能的協同

### lint → fix 流程
```
1️⃣ [[lint]] — 掃描知識庫，生成問題報告
   ↓
2️⃣ [[fix-wiki]] — 根據報告自動修復或提示人工決策
   ↓
3️⃣ [[query]] — 驗證修復後的知識連通性
```

### 工作流推薦
```
每週維護：
1. /lint — 執行周度健康檢查
2. /fix-wiki — 修復發現的問題
3. /lint — 驗證修復結果
4. [[wiki/log.md]] — 檢查操作日誌
```

## 注意事項

- ✅ 執行修復前，始終詢問需要人工決策的項
- ✅ 永遠保留操作日誌，便於回溯
- ✅ 修復知識衝突時，保留兩種觀點的記載（至少在備忘中）
- ❌ 不要無警告地刪除任何頁面
- ❌ 不要盲目刪除孤兒頁面；檢查其內容後再決策
- ❌ 修復後務必執行 lint 驗證

## 故障排除

**Q: 修復後發現有問題，可以復原嗎？**
A: 檢查 `wiki/log.md`，找到該次修復的操作記載。使用 Git history 可以查看修改前的狀態。若需要復原，聯繫使用者。

**Q: 知識衝突應該如何決策？**
A: fix-wiki 技能本身不自動決策。應向使用者展示兩種觀點，並詢問：
1. 哪一種是最新/最正確的資訊？
2. 是否可以合併成新的統一理解？
3. 如果無法決策，暫時保留衝突標記，稍後重新審視。

**Q: 孤兒頁面是否應該刪除？**
A: 不自動刪除。fix 優先嘗試重連（在其他頁面新增反向連結）。若確實無法連結（內容過時、無關），詢問使用者是否保留或刪除。
