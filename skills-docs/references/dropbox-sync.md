# Dropbox 同步與檔案衝突處理解決方案

在使用 Claude Code 或自動化腳本大規模修改 Obsidian 知識庫時，由於 Dropbox 的即時同步機制，可能會遇到「檔案被佔用」或「產生衝突副本」的問題。以下是建議的解決流程。

---

## 1. 核心解決方案：暫停同步 (Pause Syncing)

當你預計要執行「大規模更名」、「全域連結替換」或一次性「攝入大量檔案 (Batch Ingest)」時，**強烈建議先手動暫停 Dropbox 同步**。

- **操作步驟**：點擊系統工作列的 Dropbox 圖示 ⮕ 個人頭像 ⮕ **暫停同步 (Pause syncing)**。
- **好處**：防止腳本在寫入檔案時，Dropbox 同時在讀取/鎖定檔案進行上傳。
- **恢復**：操作完成後，再恢復同步，讓 Dropbox 一次處理所有變更。

---

## 2. 處理檔案衝突 (Conflicted Copies)

如果已經產生了 `... (Conflicted Copy ...) .md` 的檔案：

### A. 自動化清理腳本
你可以要求我執行以下指令來清理重複的衝突副本（以保留原始檔案為優先）：

```powershell
# 刪除 wiki 目錄下所有包含 "Conflicted Copy" 的檔案
Get-ChildItem -Path "wiki/" -Recurse -Filter "*Conflicted Copy*" | Remove-Item -Force
```

### B. 手動檢核
如果衝突發生在 `index.md` 或 `log.md` 等核心索引檔案，請先確認內容是否正確，再刪除副本。

---

## 3. 開發/編輯時的建議

1. **關閉佔用程式**：在執行大規模自動化操作時，暫時關閉 **Obsidian** 或 **VS Code**，因為這些 IDE 的自動儲存功能也會頻繁鎖定檔案。
2. **分批執行**：如果 `raw/` 下有數百個檔案，建議分批攝入（例如每次 10 個），而非一次全部執行。
3. **檢查檔案剩餘空間**：確保 Dropbox 空間充足，否則同步會卡死導致檔案損毀。

---

## 4. 進階設定：忽略特定的臨時檔案

如果 Dropbox 同步了 `.git` 或 `.gemini` 等隱藏資料夾，會浪費大量帶寬並造成衝突。

- 請在 Dropbox 設定中，將 `.git/` 和 `.gemini/`（或其他本地快取資料夾）設定為**「不轉移/僅限本地」**或使用 Dropbox 的 `attr` 指令標記。
