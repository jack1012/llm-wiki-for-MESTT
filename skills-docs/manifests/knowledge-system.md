## 🎯 各 Skill 用途

### 1. **alloc** — 文件分派與預處理

**功能**：在 ingest 之前，對 inbox/ 中的待分類檔案進行名稱標準化、屬性識別、並自動分派到 raw/ 對應的子目錄。

**核心能力**：
- 🔧 **修復檔案名稱** — 移除特殊字符、標準化空白
- 🔍 **掃描與識別** — 讀取內容，判定語言（english/中文）和類型（articles/papers/transcripts/book）
- 📂 **自動分派** — 根據識別結果移動到 raw/01-articles/, raw/02-papers/ 等

**觸發場景**：
- 「整理 inbox/ 中的檔案」
- 「準備攝取這些檔案」
- 「分類這些資料」
- `/alloc`

**工作流**：
```
inbox/File_Name.pdf
  ↓ (fix-name: 移除特殊字符)
  ↓ (scan: 識別 type=papers, language=english)
  ↓ (move)
raw/02-papers/File_Name.pdf
```

**最佳實踐**：
- ✅ 定期執行 alloc 保持 inbox/ 整潔
- ✅ 使用標準化檔案名稱便於後續處理
- ✅ 若無法自動判定類型，alloc 會暫停並詢問
- ❌ 不要手動移動檔案，應使用 alloc 確保一致性

**文檔**：
- 完整說明：[[skills-docs/guide.md#alloc]]
- 官方定義：[.claude/skills/alloc/SKILL.md]

### 2. **ingest** — 知識編譯與提煉

**功能**：將 raw/ 目錄下的原始資料編譯到 wiki/ 中，自動提煉摘要、實體、概念，並歸檔原始檔。

**核心能力**：
- 📖 **讀取原始檔** — 支援 .md, .pdf, .txt 等格式
- ✏️ **提煉核心** — 自動摘要、提取實體和概念
- 🌐 **知識網路化** — 建立 wiki/sources/, wiki/entities/, wiki/concepts/ 頁面
- 📝 **維護索引** — 自動更新 wiki/index.md 和 wiki/log.md
- 📦 **歸檔處理** — 完成後移動到 raw/09-archive/

**觸發場景**：
- 「攝取這篇文章」
- 「匯入這份資料到知識庫」
- 「處理待處理檔案」
- `/ingest`

**工作流**：
```
1️⃣ raw/02-papers/paper.pdf
  ↓
2️⃣ 讀取內容，提煉摘要、實體、概念
  ↓
3️⃣ 建立 wiki/sources/summary-paper.md
4️⃣ 建立 wiki/entities/EntityName.md
5️⃣ 建立 wiki/concepts/ConceptName.md
  ↓
6️⃣ 更新 wiki/index.md, wiki/log.md
  ↓
7️⃣ raw/09-archive/paper.pdf （歸檔）
```

**最佳實踐**：
- ✅ 先執行 alloc 確保檔案在正確位置
- ✅ 定期執行 ingest 處理待處理檔案
- ✅ 遇到衝突時暫停，詢問如何決策
- ✅ 檢查 wiki/log.md 確認操作記載
- ❌ 不要修改 raw/ 下的檔案內容
- ❌ 不要一次性處理過多檔案，分批執行

**文檔**：
- 完整說明：[[skills-docs/guide.md#ingest]]
- 官方定義：[.claude/skills/ingest/SKILL.md]

### 3. **lint** — 知識庫健康檢查

**功能**：全域掃描 wiki/ 目錄，檢測死連結、孤兒頁面、未同步索引、知識衝突等問題，生成健康度報告。

**核心能力**：
- 🔗 **死連結檢測** — 找出指向不存在頁面的 wikilinks
- 👻 **孤兒頁面檢測** — 找出從未被引用的頁面
- 📋 **索引同步檢查** — 確認 index.md 與實際頁面一致
- ⚠️ **衝突識別** — 尋找包含 `## 知識衝突` 的頁面

**觸發場景**：
- 「檢查知識庫狀況」
- 「執行健康檢查」
- 「掃描死連結」
- `/lint`

**輸出**：結構化報告（綠/黃/紅燈）

**最佳實踐**：
- ✅ 定期（每週）執行 lint 維持知識庫健康
- ✅ 遵循報告的建議，使用 fix 執行修復
- ✅ 在 index.md 中保持頁面註冊完整
- ❌ 不要無視衝突警告
- ❌ 不要修改 lint 報告，應執行 fix 修復

**文檔**：
- 完整說明：[[skills-docs/guide.md#lint]]
- 官方定義：[.claude/skills/lint/SKILL.md]

---

### 4. **fix** — 知識庫自動修復

**功能**：根據 lint 巡檢報告，自動執行或輔助修復死連結、孤兒頁面、索引不同步等問題。

**核心能力**：
- 🔧 **死連結修復** — 刪除或修正不存在的連結
- 🔌 **孤兒重連** — 在其他頁面新增反向連結重連孤兒頁面
- 📝 **索引同步** — 新增缺失的 index.md 條目
- ⚖️ **衝突決策** — 輔助使用者決策知識衝突

**觸發場景**：
- 「修復知識庫」
- 「解決死連結」
- 「同步索引」
- `/fix-wiki`

**工作流**：
```
1️⃣ lint → 生成問題報告
  ↓
2️⃣ fix → 自動修復可修復項、詢問需決策項
  ↓
3️⃣ 使用者確認決策
  ↓
4️⃣ 更新 wiki/ 頁面和 index.md, log.md
  ↓
5️⃣ lint → 驗證修復結果
```

**最佳實踐**：
- ✅ 執行 lint 前先執行 fix，清理上次發現的問題
- ✅ 對於需決策的項（如知識衝突），仔細思考後再確認
- ✅ 修復後重新執行 lint 驗證
- ✅ 檢查 wiki/log.md 確認修復記載
- ❌ 不要盲目刪除孤兒頁面，優先嘗試重連
- ❌ 不要自動決策知識衝突，應向使用者詢問

**文檔**：
- 完整說明：[[skills-docs/guide.md#fix]]
- 官方定義：[.claude/skills/fix-wiki/SKILL.md]

### 5. **query** — 本地知識查詢

**功能**：在本地 wiki/ 知識庫中回答使用者提問，自動定位相關頁面並以雙向連結格式引用。

**核心能力**：
- 🔍 **知識搜尋** — 根據提問關鍵字搜尋 wiki/ 相關頁面
- 📖 **深度閱讀** — 讀取相關頁面並理解上下文
- 🔗 **連結引用** — 以 wikilink 格式引用來源
- ⚠️ **邊界聲明** — 知識庫缺失時明確聲明

**觸發場景**：
- 「我的筆記中有關於 X 的內容嗎？」
- 「查詢我之前的決定」
- 「我的知識庫中怎麼說的？」
- `/query <問題>`

**工作流**：
```
1️⃣ 讀取 wiki/index.md 定位相關頁面
  ↓
2️⃣ 深度閱讀相關頁面
  ↓
3️⃣ 綜合分析、提取答案
  ↓
4️⃣ 以 [[wikilink]] 格式引用來源回答
```

**最佳實踐**：
- ✅ 優先使用本地知識庫回答
- ✅ 若知識庫缺失，明確聲明後提供通用知識
- ✅ 在回答中使用 wikilinks 引用來源
- ✅ 定期檢查 lint 報告確保知識庫可查詢性
- ❌ 不要憑模型記憶回答，必須查詢本地知識庫
- ❌ 不要制造虛假的 wikilinks 引用

**文檔**：
- 完整說明：[[skills-docs/guide.md#query]]
- 官方定義：[.claude/skills/query/SKILL.md]

## ✨ 完整工作流

### 標準工作流：檔案 → 知識庫

```
1️⃣ 新檔案到達
   inbox/file.pdf

2️⃣ [[alloc]] — 分派與預處理
   /alloc
   → 修復名稱、識別屬性
   → 移動到 raw/02-papers/

3️⃣ [[ingest]] — 知識編譯
   /ingest
   → 讀取內容、提煉核心
   → 建立 wiki 頁面
   → 歸檔原始檔

4️⃣ [[lint]] — 健康檢查
   /lint
   → 掃描知識庫問題
   → 生成報告

5️⃣ [[fix-wiki]] — 自動修復
   /fix-wiki
   → 修復發現的問題
   → 更新索引和日誌

6️⃣ [[query]] — 知識查詢
   /query
   → 驗證新知識可查詢
```

### 定期維護工作流（每週）

```
1️⃣ 週一：掃描 inbox/，執行 alloc
2️⃣ 週二-四：執行 ingest，處理待處理檔案
3️⃣ 週五：執行 lint，檢查健康狀況
4️⃣ 週五：執行 fix，修復發現的問題
5️⃣ 每日：執行 query，驗證知識可達性
```
