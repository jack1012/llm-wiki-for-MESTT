### 1️⃣ 知識庫核心系統（5 個）— LLM Wiki 專用

用於整理、編譯、維護、查詢你的知識庫。

| Skill | 類型 | 狀態 | 核心功能 |
|-------|------|------|---------|
| **[[alloc]]** | 自訂 | ✅ | 文件分派與預處理（inbox → raw/） |
| **[[ingest]]** | 自訂 | ✅ | 知識庫攝入（原始資料 → wiki） |
| **[[lint]]** | 自訂 | ✅ | 健康檢查（死連結、孤兒頁面） |
| **[[fix-wiki]]** | 自訂 | ✅ | 知識庫修復（自動修復問題） |
| **[[query]]** | 自訂 | ✅ | 知識查詢（本地 wiki 問答） |

**一句話說明**：
- 🔵 **alloc** — 整理 inbox/ 檔案，修復名稱、識別屬性、自動分派到 raw/
- 🔵 **ingest** — 把論文、文章、網頁內容快速編譯到知識庫
- 🔵 **lint** — 定期掃描知識庫，找出死連結、孤兒頁面、索引不同步
- 🔵 **fix** — 根據 lint 報告自動修復或輔助決策
- 🔵 **query** — 在你的知識庫中提問，獲得基於本地知識的答案

**常見工作流**：`alloc → ingest → lint → fix → query`

## 知識庫核心系統

### 1. [[alloc]] — 文件分派與預處理

**類型**：🔵 自訂技能  
**狀態**：✅ 活躍  
**版本**：1.0  
**最後更新**：2026-04-16

#### 功能說明

在 ingest 之前，對 inbox/ 中的待分類檔案進行名稱標準化、屬性識別、並自動分派到 raw/ 對應的子目錄。確保檔案準備就緒後再進行知識編譯。

#### 核心能力

- 🔧 修復檔案名稱（移除特殊字符、標準化空白）
- 🔍 掃描與識別（讀取內容，判定語言和類型）
- 📂 自動分派（根據類型移動到 raw/01-articles/, raw/02-papers/ 等）
- 📊 生成分派報告（成功、失敗、需確認統計）

#### 觸發方式

```
# 方式 1：掃描整個 inbox/
/alloc

# 方式 2：處理特定檔案
/alloc inbox/file.pdf

# 方式 3：自然語言觸發
「整理一下 inbox/ 中的檔案」
「準備攝取這些資料」
「分類這些檔案」
```

#### 使用範例

**範例 1：批量分派 inbox/ 中的檔案**
```
操作：/alloc

流程：
1. 掃描 inbox/ 下所有檔案
2. 修復名稱（移除特殊字符）
3. 掃描內容，識別語言和類型
4. 自動分派到相應目錄

結果：
✅ 成功分派：7 個檔案
  - 2 個 PDF → raw/02-papers/
  - 3 個 MD 文章 → raw/01-articles/
  - 2 個轉錄文稿 → raw/03-transcripts/
⚠️ 需確認：1 個（無法自動判定類型）
```

**範例 2：修復單個檔案名稱並分派**
```
操作：/alloc inbox/[論文]2026-Nature AI研究%最新版(v2.1).pdf

流程：
1. 修復名稱 → 論文2026-Nature_AI研究最新版v2.1.pdf
2. 掃描內容 → type: papers, language: 中文
3. 分派 → raw/02-papers/論文2026-Nature_AI研究最新版v2.1.pdf

結果：
✅ 已分派：1 個檔案 → raw/02-papers/
```

#### 最佳實踐

✅ **DO**
- 定期執行 `alloc` 保持 inbox/ 整潔
- 對於無法自動判定類型的檔案，仔細檢查內容後手動指定
- 使用標準化檔案名稱便於後續引用
- 查看分派報告確保所有檔案正確分類

❌ **DON'T**
- 不要手動修改 inbox/ 中的檔案，應使用 alloc
- 不要跳過無法判定的檔案，應進行人工確認
- 不要盲目修改檔案副檔名

#### 進階用法

**與其他 skills 的銜接：**
```
1. [[alloc]] skill
   → 整理並分派新檔案到 raw/

2. [[ingest]] skill
   → 編譯到知識庫

3. [[lint]] skill
   → 檢查知識庫狀況

4. [[fix-wiki]] skill
   → 修復發現的問題
```

### 2. [[ingest]] — 知識庫攝入

**類型**：🔵 自訂技能  
**狀態**：✅ 活躍  
**版本**：1.0  
**最後更新**：2026-04-15

#### 功能說明

將 `raw/` 目錄下的原始資料（文章、論文、轉錄文案）編譯到 `wiki/` 知識庫中。支援自動生成摘要頁面、概念頁面、實體頁面，並自動更新索引。

#### 核心能力

- 📖 讀取多種格式（Markdown、PDF、文本）
- 🔍 提取核心主旨和關鍵詞
- 📝 生成來源摘要頁面（`wiki/sources/`）
- 🏷️ 建立/更新實體和概念頁面（`wiki/entities/`、`wiki/concepts/`）
- 🔗 自動建立雙向連結
- 📋 更新 index.md 和 log.md

#### 觸發方式

```
# 方式 1：掃描所有未歸檔檔案
/ingest

# 方式 2：處理特定檔案
/ingest raw/02-papers/my-paper.pdf

# 方式 3：自然語言觸發
「幫我攝取這個檔案」
「將這篇論文匯入知識庫」
「處理 raw/ 目錄下的資料」
```

#### 使用範例

**範例 1：攝入單個 PDF 論文**
```
操作：/ingest raw/02-papers/my-research-paper.pdf

流程：
1. 讀取 PDF 內容
2. 提取：主旨、實體（作者、機構）、概念（理論、方法）
3. 生成 wiki/sources/summary-my-research-paper.md
4. 建立/更新相關概念頁面
5. 自動歸檔原始檔到 raw/09-archive/
6. 更新 wiki/index.md 和 wiki/log.md

結果：
✅ 新增 1 個 source
✅ 新增/更新 3-5 個概念
✅ 建立 2-3 個新實體頁面
```

**範例 2：批量攝入所有待處理檔案**
```
操作：/ingest

流程：
1. 掃描 raw/ 下所有子目錄（除 09-archive/）
2. 發現 7 個 PDF、3 個 Markdown 檔案
3. 依次處理，更新索引
4. 生成統計報告

結果：
✅ 已處理：10 個檔案
✅ 新增：6 個 sources + 18 個 concepts + 5 個 entities
✅ 發現衝突：2 個（已暫停，等待人工決策）
```

#### 最佳實踐

✅ **DO**
- 定期執行 `ingest`，保持 raw/ 目錄清潔
- 處理論文時，使用 5-Pass 框架進行深度分析
- 衝突出現時，仔細檢查並手動決策
- 為每個來源記錄 frontmatter 的 `sources` 欄位

❌ **DON'T**
- 不要手動修改 raw/ 目錄中的檔案
- 不要跳過衝突處理直接覆蓋
- 不要在 ingest 中途中斷（可能導致狀態不一致）
- 不要忘記查看 wiki/log.md 記錄

#### 進階用法

**結合其他 skills 的工作流：**
```
1. [[pdf]] skill
   → 從 PDF 提取表格和圖表

2. [[ingest]] skill
   → 編譯提取的內容到 wiki/

3. [[skill-creator]] skill
   → 建立「論文分析工作流」

4. [[query]] skill
   → 查詢和綜合知識
```

### 3. [[lint]] — 知識庫健康檢查

**類型**：🔵 自訂技能  
**狀態**：✅ 活躍  
**版本**：1.0  
**最後更新**：2026-04-15

#### 功能說明

全域掃描 `wiki/` 目錄，偵測：
- 💀 死連結（引用不存在的頁面）
- 🏝️ 孤兒頁面（沒有任何連結指向它）
- 📊 未同步索引（檔案存在但 index.md 未記錄）
- ⚠️ 知識衝突（同一概念有多個相互矛盾的說法）

#### 核心能力

- 🔍 遍歷所有 wiki 檔案
- 🔗 檢驗雙向連結的有效性
- 📋 驗證 index.md 的完整性
- 🎯 生成結構化報告
- 🛠️ 提供修復建議

#### 觸發方式

```
# 方式 1：完整掃描
/lint

# 方式 2：自然語言觸發
「檢查知識庫狀態」
「執行健康檢查」
「掃描我的 wiki」
「有沒有死連結？」
```

#### 使用範例

**範例 1：定期健康檢查**
```
操作：/lint

輸出報告：

🩺 知識庫健康體檢報告 — 2026-04-16

✅ 綠燈項
- 總頁面數：52 個（↑ 新增 6 個）
- 平均連結數：4.2 個/頁
- 引用覆蓋率：94%

⚠️ 黃燈項
- 發現 3 個孤兒頁面：
  • [[Orphan Page 1]] — 無任何連結指向
  • [[Orphan Page 2]] — 無任何連結指向
  • [[Orphan Page 3]] — 無任何連結指向
  
  建議：新增關聯連結或刪除

- 發現 2 個未同步索引：
  • [[New Concept 1]] — 檔案存在但未在 index.md 註冊
  • [[New Concept 2]] — 檔案存在但未在 index.md 註冊
  
  建議：自動修復或手動確認

❌ 紅燈項
- 發現 5 個死連結：
  • [[Frame of Reference]] → [[Non-existent Page]]
  • [[Mathematical Crisis]] → [[Undefined Concept]]
  ...
  
  建議：檢查拼寫或建立缺失頁面

- 存在 1 個未解決的知識衝突：
  • [[Mathematical Beliefs]] — 靜態 vs 動態定義衝突
  
  建議：閱讀衝突區塊並手動決策

🛠️ 下一步行動
1. 修復 5 個死連結？ → 自動修復 / 手動檢查
2. 解決 3 個孤兒頁面？ → 新增連結 / 刪除頁面
3. 同步 2 個未註冊頁面？ → 自動同步 / 手動確認
```

**範例 2：修復工作流**
```
操作：lint 報告後進行修復

步驟 1：自動修復未同步索引
→ system 更新 index.md，新增遺漏頁面

步驟 2：手動檢查死連結
→ 逐一驗證每個死連結，決定修復或刪除

步驟 3：解決孤兒頁面
→ 決定新增連結或刪除頁面

步驟 4：處理知識衝突
→ 閱讀雙方說法，標記為「待研究」或合併內容

步驟 5：重新運行 lint
→ 確認所有問題已解決
```

#### 最佳實踐

✅ **DO**
- 每週運行一次 `lint`
- 及時處理紅燈項（死連結）
- 定期檢查黃燈項（孤兒頁面、未同步索引）
- 記錄修復過程到 wiki/log.md

❌ **DON'T**
- 不要忽視紅燈項
- 不要盲目刪除孤兒頁面（可能有用）
- 不要用靜默覆蓋解決知識衝突
- 不要跳過修復直接新增內容

### 4. [[fix-wiki]] — 知識庫修復

**類型**：🔵 自訂技能  
**狀態**：✅ 活躍  
**版本**：1.0  
**最後更新**：2026-04-16

#### 功能說明

根據 lint 巡檢報告，自動執行或輔助修復知識庫中的問題（死連結、孤兒頁面、索引不同步、知識衝突）。完成修復後自動更新 index.md 和 log.md。

#### 核心能力

- 🔧 死連結修復（刪除或修正不存在的連結）
- 🔌 孤兒頁面重連（在其他頁面新增反向連結）
- 📝 索引同步（新增缺失的 index.md 條目）
- ⚖️ 知識衝突決策（輔助使用者決策）
- 📋 自動更新 index.md 和 log.md

#### 觸發方式

```
# 方式 1：執行所有建議修復（需逐項確認）
/fix-wiki

# 方式 2：修復特定類型
/fix-wiki deadlink
/fix-wiki orphan
/fix-wiki unsync
/fix-wiki conflict

# 方式 3：自動執行所有可自動修復的項
/fix-wiki --auto_confirm

# 方式 4：自然語言觸發
「修復知識庫」
「解決死連結」
「同步索引」
```

#### 使用範例

**範例 1：根據 lint 報告修復全部問題**
```
操作：/lint → 發現問題 → /fix-wiki

流程：
1. lint 發現：
   ❌ 5 個死連結
   ⚠️ 3 個孤兒頁面
   ⚠️ 2 個未同步索引

2. fix 執行修復：
   ✅ 死連結修復：5 個已刪除或修正
   ⚠️ 孤兒頁面：詢問是否保留或重連
   ✅ 索引同步：2 個已新增到 index.md

3. 更新日誌：
   wiki/log.md 追加修復記載
```

**範例 2：僅修復死連結**
```
操作：/fix-wiki deadlink

流程：
1. 掃描所有頁面中的死連結
2. 嘗試模糊匹配（如 [[RAG]] 可能應該是 [[RAG架構]]）
3. 提議修復或刪除
4. 使用者確認後執行
5. 更新頁面和 log.md

結果：
✅ 修復：3 個死連結已更正
⚠️ 待確認：2 個（無法自動決策）
```

#### 最佳實踐

✅ **DO**
- 在執行 ingest 後運行 lint，再執行 fix
- 對於需決策的項（如知識衝突），仔細思考後再確認
- 修復後重新執行 lint 驗證修復結果
- 檢查 wiki/log.md 確認修復記載

❌ **DON'T**
- 不要盲目刪除孤兒頁面，優先嘗試重連
- 不要無警告地刪除任何頁面
- 不要自動決策知識衝突，應詢問使用者
- 不要在不理解問題的情況下執行自動修復

#### 進階用法

**完整的知識庫維護工作流：**
```
1. [[alloc]] skill
   → 整理 inbox/ 檔案

2. [[ingest]] skill
   → 編譯到知識庫

3. [[lint]] skill
   → 掃描問題

4. [[fix-wiki]] skill
   → 修復問題

5. [[lint]] skill（驗證）
   → 確認問題已解決

6. [[query]] skill
   → 驗證知識可查詢
```

### 5. [[query]] — 知識庫查詢

**類型**：🔵 自訂技能  
**狀態**：✅ 活躍  
**版本**：1.0  
**最後更新**：2026-04-15

#### 功能說明

在本地 wiki 知識庫中回答使用者問題。首先讀取 `wiki/index.md` 定位相關頁面，再深度閱讀，最後以雙向連結引用格式回答。禁止依賴模型記憶，必須從本地知識庫中尋找證據。

#### 核心能力

- 🔍 全文搜尋 wiki 內容
- 📖 深度閱讀相關頁面
- 🧠 綜合多個來源的知識
- 🔗 自動生成引用連結
- ⚠️ 區分本地知識 vs 通用知識

#### 觸發方式

```
# 方式 1：/query 命令
/query 數學信念對大學過渡的影響是什麼？

# 方式 2：自然語言觸發
「根據我的筆記，... 是什麼？」
「我之前記錄過有關 ... 的內容嗎？」
「我的知識庫中有 ... 相關的資訊嗎？」
```

#### 使用範例

**範例 1：查詢概念**
```
提問：「什麼是數學信念過度懸掛？」

query 流程：
1. 掃描 wiki/index.md，定位相關頁面：
   - [[Mathematical Beliefs]]
   - [[Beliefs-overhang]]
   - [[Static vs Dynamic Mathematics Beliefs]]

2. 深度閱讀這些頁面

3. 綜合回答：

根據你的知識庫，**數學信念過度懸掛**（Beliefs-overhang）是指：

學生持有的靜態數學信念（認為數學能力是固定的）與大學實際要求的動態思維之間的落差。這種不匹配會導致：
- 成就滿意度下降（[[Mathematical Beliefs]] — 第 2 節）
- 首學期動態信念下降現象（[[summary-that-wasnt-the-math-i-wanted]]）
- 過渡期困境加重（[[The Mathematical Crisis in Secondary-Tertiary Transition]]）

相關研究：
- Ng (2023) 的 IJMEST 論文重點研究了這一現象
- IRME 研究工具可用於測量信念類型

延伸閱讀：[[Mathematical Resilience]]、[[Attribution Theory]]
```

**範例 2：比較不同概念**
```
提問：「我的知識庫中，情感變項和數學信念之間的關係是什麼？」

query 綜合回答：

根據你的知識庫，情感變項和數學信念之間存在複雜的相互作用：

1️⃣ **信念影響情感**
   - 靜態信念 → 面對困難時產生低自信感
   - 導致負面情感（焦慮、挫敗）
   - 最終影響學業興趣和成就

2️⃣ **情感反饋強化信念**
   - 負面情感體驗 → 強化靜態信念
   - 形成「低能力循環」

3️⃣ **關鍵預測變項**
   - [[Affective Variables]] 中記錄：
   - 興趣 > 信念 > 滿意度（預測留存率）
   - 滿意度是中介變項，影響成就與留存

相關文獻：
- [[Development of affect at the transition]]（核心論文）
- [[What role do student beliefs play]]（信念研究）

連結圖譜：
[[Mathematical Beliefs]] ← → [[Affective Variables]]
  ↓                        ↓
[[Attribution Theory]] ← [[Early Dropout]]
```

#### 最佳實踐

✅ **DO**
- 先搜索本地知識庫，確保有證據支持
- 使用雙向連結（[[]]）引用相關頁面
- 引用具體的論文或筆記區域
- 說明答案的置信度（基於多少來源）

❌ **DON'T**
- 不要依賴模型記憶回答
- 不要誤導說本地沒有內容（未充分搜索）
- 不要遺漏引用來源
- 不要混淆本地知識和通用知識

## 文檔與 PDF Skills

### 4. [[pdf]] — PDF 處理與表單填寫

**類型**：🟢 官方技能（Anthropic）  
**狀態**：✅ 活躍  
**版本**：1.0  
**來源**：https://github.com/anthropics/skills/tree/main/skills/pdf

#### 功能說明

完整的 PDF 處理工具包。支援讀取、提取、轉換、合併、拆分、表單填寫等操作。內含 8 個 Python 指令碼用於高級操作。

#### 核心能力

- 📖 **讀取和提取**
  - 文本提取（保留或去除格式）
  - 表格提取（自動格式化為表格）
  - 圖像提取
  - 元數據提取

- 🔗 **合併與拆分**
  - 合併多個 PDF
  - 按頁面拆分
  - 按範圍拆分
  - 重新排列頁面

- 🎨 **頁面操作**
  - 旋轉頁面
  - 添加水印（文本或圖像）
  - 調整頁面大小
  - 裁剪頁面

- 📝 **表單處理**
  - 識別可填充欄位
  - 自動填寫表單
  - 驗證表單結構
  - 批註合併

- 🔐 **安全性**
  - 加密 PDF（設定密碼）
  - 解密 PDF

- 🔍 **高級功能**
  - OCR（掃描文檔文字辨識）
  - 轉換為圖像
  - 提取邊界框信息

#### 觸發方式

```
# 自動觸發
「讀取這個 PDF」
「提取 PDF 表格」
「合併這些 PDF」
「填寫 PDF 表單」
「對掃描 PDF 進行 OCR」
「拆分 PDF 為單個頁面」

# 或提及 .pdf 檔案
「我有一個 document.pdf」
```

#### 使用範例

**範例 1：提取論文表格**
```
提問：「讀取 my-paper.pdf，提取所有表格」

pdf skill 執行：
1. 開啟 PDF 檔案
2. 逐頁掃描，識別表格
3. 自動格式化為 Markdown 表格
4. 提取相關文本內容

結果：
✅ 發現 3 個表格
表格 1：研究樣本統計
表格 2：變項相關矩陣
表格 3：迴歸係數

[Markdown 表格輸出]

可進一步操作：
→ 複製到 Excel
→ 使用 [[ingest]] 匯入知識庫
→ 使用 [[skill-creator]] 建立表格處理工作流
```

**範例 2：批量 PDF 合併**
```
提問：「合併 paper1.pdf, paper2.pdf, paper3.pdf」

pdf skill 執行：
1. 開啟三個 PDF
2. 逐個複製所有頁面
3. 輸出為 merged.pdf

結果：
✅ 合併完成
📊 原始檔案：
   - paper1.pdf (15 頁)
   - paper2.pdf (12 頁)
   - paper3.pdf (10 頁)

📄 輸出檔案：merged.pdf (37 頁)
```

**範例 3：表單自動填寫**
```
提問：「填寫這個 PDF 表單，使用以下數據：
   Name: John Doe
   Email: john@example.com
   Date: 2026-04-16」

pdf skill 執行：
1. 識別表單結構
2. 定位可填充欄位
3. 逐個填寫數據
4. 輸出為 completed-form.pdf

結果：
✅ 表單填寫完成
✅ 所有欄位已驗證
📄 輸出：completed-form.pdf
```

#### 最佳實踐

✅ **DO**
- 在提取前確認 PDF 格式（掃描 vs 數位版）
- 驗證提取的表格結構
- 為重要 PDF 保留備份
- 使用 OCR 處理掃描 PDF

❌ **DON'T**
- 不要對已加密 PDF 進行操作（先解密）
- 不要假設所有 PDF 都能完美提取
- 不要忘記驗證表單欄位
- 不要對原始檔案進行修改

### 5. [[defuddle]] — 網頁內容提取

**類型**：🟢 官方技能（Anthropic）  
**狀態**：✅ 活躍  
**版本**：1.0  
**來源**：https://github.com/kepano/obsidian-skills

#### 功能說明

從網頁提取乾淨的 Markdown 內容，自動移除導航、廣告、雜亂內容，只保留核心文章內容。節省 token 開銷，提高內容品質。

#### 核心能力

- 🧹 **內容清潔**
  - 移除導航菜單
  - 移除廣告和追蹤碼
  - 移除頁腳和非核心內容
  - 保留文章主體

- 📝 **格式轉換**
  - 網頁 HTML → 清潔 Markdown
  - 保留標題、段落、列表結構
  - 保留圖片連結
  - 保留表格

- 💾 **Token 優化**
  - 減少不必要文字
  - 優化內容大小
  - 加速後續處理

#### 觸發方式

```
# 自動觸發
「提取這個網頁內容」
「清潔並轉換網頁為 Markdown」
「從 URL 提取文章」

# 或使用特定命令
/defuddle https://example.com/article
```

#### 使用範例

**範例 1：從研究文章提取**
```
提問：「提取 https://example.com/research-article 的內容」

defuddle 執行：
1. 讀取網頁 HTML
2. 識別主要內容區域
3. 移除雜亂元素
4. 轉換為 Markdown

結果：
✅ 提取成功
📄 原始檔案大小：250 KB
📄 清潔後大小：35 KB
⚡ Token 節省：約 60%

[清潔後的 Markdown 內容]

可進一步操作：
→ 複製到 wiki/ 中
→ 使用 [[ingest]] 編譯到知識庫
```

#### 最佳實踐

✅ **DO**
- 對長篇文章使用 defuddle
- 驗證提取的內容完整性
- 檢查圖片連結是否正常
- 保存原始 URL 作為來源

❌ **DON'T**
- 不要對 PDF 使用 defuddle（用 [[pdf]] 代替）
- 不要忽視提取後的質量檢查
- 不要丟失原始 URL
- 不要對動態網頁期望完美結果

## 工作流推薦

### 工作流 1：完整論文處理

```
目標：從 PDF 論文到知識庫中的完整記錄

步驟 1：[[pdf]] — 讀取論文，提取表格、圖像

   ↓
步驟 3：[[skill-creator]] — 建立「論文分析技能」
   ↓
步驟 4：[[query]] — 查詢和綜合知識
   ↓
步驟 5：[[lint]] — 驗證完整性
```

